XHTML, deel 2

XHTML versus HTML

Correctly nested elements

Omdat XHTML een vorm van XML is, moet een XHTML document well-formed zijn, dus:

<b>Close the italics tag <i>first</i></b>
<b>This line is <i>not</b>Well formed!is</i>

Daarnaast zijn o.a. de volgende nesting restricties van kracht:

  • The <a> tag cannot contain another <a> tag
  • The <pre> tag cannot contain <img>, <object>, <big>, <small>, <sub> or <sup> tags
  • The <button> tag cannot contain <input>, <select>, <textarea>, <label>, <button>, <form>, <fieldset>, <iframe> or <isindex> tags
  • The <form> tag cannot contain other <form> tags

End tags

Iedere tag - met of zonder andere tags of content - moet een corresponderende end tag hebben. Een missing end tag is een error en maakt het document noncompliant.

Gelukkig is er vaak een shortcut-notatie mogelijk, zoals in het volgende voorbeeld. In plaats van <br></br> mag je ook schrijven <br />, of <img src="plaatje.gif" />

Case-Sensitivity

XHTML is case-sensitive voor alle tag- en attribuut-namen. Dus in een XHTML document zijn <a> en <A> verschillende tags.
Bovendien legt XHTML vast dat alle bestaande HTML tags en attributen uit lowercase letters moeten bestaan. Uppercase tag of attribuut namen zijn geen geen geldige XHTML tags of attributen.

Quoted Attribute Values

XHTML verplicht je om attribuutwaardes - ook de numerieke - tussen dubbele quotes te zetten.
In HTML kon je doen wat je wilde m.u.v. de situatie waarin de attribuutwaarde whitespace of andere speciale karakters bevatte.
Om XHTML-compliant te zijn moet je <table rows="3"> gebruiken!

Explicit Attribute Values

In HTML is er een klein aantal attributen die geen waarden hebben. Alleen de aanwezigheid van het attribuut laat de tag zich anders gedragen. Deze attributen gedragen zich vaak als een aan/uit switch.
Een voorbeeld is het checked attribuut dat b.v. bij een checkbox gebruikt kan worden.

In XHTML moet ieder attribuut een waarde hebben. Het checked attribuut gebruik je dan als volgt: checked="checked"

De attributen id en name

HTML gebruikt het name attribuut bij b.v. de <a> tag om een zgn. fragment identifier in het document aan te maken. Deze kan dan gebruikt worden om naar een betreffende plek in het document te springen. Later is het name attribuut ook aan andere tags zoals het <frame> en <img> tag toegeveogd.
Met HTML 4.0 heeft het W3C ook het id attribuut aan vrijwel iedere tag toegevoegd, en wel met dezelfde functie als het name attribuut.

XHTML heeft een sterke voorkeur voor het id attribuut. Omdat het gebruik van HTML 4.0 inmiddels wijd verbreid is, zou je moeten proberen het name attribuut waar mogelijk te vermijden en in plaats daarvan het id attribuut moeten gaan gebruken om namen aan elementen te koppelen, of minstens beide attributen (met dezelfde waardes) op te nemen.

XHTML 1.1

In mei 2001 heeft het W3C een aangepaste XHTML standaard vrijgegeven: XHTML 1.1.
Waar standaarden meestal een uitbreiding zijn van eerdere versies, zet XHTML 1.1 een stap in de omgekeerde richting: XHTML 1.1 begint met de XHTML 1.0 strict DTD en brengt daar enkele wijzigingen in aan.

Overstappen naar XHTML

Inmiddels mag duidelijke zijn dat XHTML duidelijke een pijnlijkere en minder vergevingsgezinde weg is dan HTML. Waarom die extra moeite doen dan?

Het Dusty Deck Probleem

Het Web bestaat inmiddels uit miljoenen pagina's, en je kunt er je geld op zetten dat veel daarvan officieel niet compliant zijn met enige gedefinieerde versie van HTML. Duidelijk mag ook zijn dat een nog groter deel niet XHTML-compliant is.
De vraag is zeer terecht of (al) deze pagina's ooit naar XHTML geconverteerd zullen worden: wie heeft of neemt daar de tijd voor? De bezoeker ziet het verschil niet!
Het dusty deck probleem is echter geen excuus om vanaf nu niet te beginnen met het schrijven van documenten die wel comliant zijn. Laat de oude rommel met rust, maar creëer niet nog meer conversie problemen. Een klein beetje inspanning zal helpen om je documenten over een bredere range van browsers te kunnen gebruiken in de toekomst.

Automatische Conversie

Er zijn tools die je kunnen helpen bij het omzetten van je bestaande HTML documenten naar XHTML.
Een voorbeeld is een utility genaamd Tidy, geschreven door Dave Raggett, 1 van de movers en shakers bij het W3C. Tidy kan o.a. case conversion, quoted attributes, en proper element nesting detecteren en corrigeren. Zie http://tidy.sourceforge.net voor een compleet overzicht van features en voor de meest recente versie voor de diverse platformen.

Lenient Browsers and Lazy Authors

Er is een algemene vuistregel betreffende data sharing, zeker op het internet: be lenient in what you accept and strict in what you produce.
Web browsers zijn een goed voorbeeld van lenient acceptors. Heel veel web pagina's bevatten fouten in een of andere zin. Toch gaan browsers daar meestal heel behulpzaam mee om. Sterker nog: een strenge browser heeft niet eens een fatsoenlijke overlevingskans! Vergevinsgezinde browsers doen het aantal 'slechte' pagina's echter weer toenemen ... een vicieuze cirkel!
Doorbreek deze cyclus door te alleen XHTML-compliant pagina's te maken wanneer mogelijk.

Time, Money en Standards

XHTML is ontwikkeld als een XML representatie van de HTML standaard. Het is bedoeld om de enige standaard te worden die iedereen zou moeten gebruiken om content voor het web te creeren.
Zover is het nog (lang) niet, want dit gaat uit van een ideale wereld, en het kost veel tijd en geld. Uiteindelijk zal hopelijk het verstand om te standaardiseren - en daarmee veel grotere uitwisselbaarheid te bereiken - zegevieren.

Man Versus Machine

Hoewel het voor mensen lastig is om XHTML te produceren, kunnen programma's dat in de regel heel goed. Het aantal web-authoring tools neemt nog steeds toe, evenals het aantal pagina's dat daarmee gemaakt wordt en die dus compleet XHTML-compliant zou moeten zijn.
Iets vergelijkbaars geldt voor de makers van andere software die nu op de een of andere manier HTML genereren of verwerken. Ook deze software zou eenvoudig(er) XHTML moeten kunnen verwerken.
